# Инструкция для AI‑агента

Цель: поддерживать этот репозиторий как **Obsidian‑vault в Git** для личных/рабочих задач, заметок, проектов и коучинга по целям.

## Золотые правила
- **Минимизируй дифф**: не делай массового переформатирования, не меняй стиль без необходимости.
- **Не удаляй данные**: если что-то устарело — помечай статусом и/или переноси в `99_Archive/`.
- **Пиши атомарно**: небольшие изменения, логические блоки, понятные заголовки.
- **Ссылки вместо дублей**: используй `[[wiki links]]` между задачами/проектами/целями.
- **ISO‑даты**: `YYYY-MM-DD` в метаданных и свойствах Tasks.

## Куда что писать
- Входящее: `Personal/00_Inbox/` или `Work/00_Inbox/`.
- Задачи: `Personal/20_Tasks/` или `Work/20_Tasks/` (1 файл = 1 задача).
- Проекты: `Personal/30_Projects/` или `Work/30_Projects/`.
- Области (responsibility): `Personal/40_Areas/` или `Work/40_Areas/`.
- Заметки/знания: `Personal/50_Notes/` или `Work/50_Notes/`.
- Цели: `Personal/60_Goals/` или `Work/60_Goals/`.
- Обзоры: `Personal/70_Reviews/` или `Work/70_Reviews/`.
- Шаблоны/правила: `00_System/`.

## Как правильно использовать Area
- `Area` = постоянная зона ответственности / повторяемый тип работы с общими правилами (гайдлайны, чеклисты, стандарты качества).
- `Project` = конкретный результат (outcome) с финалом.
- `Task` = конкретное действие/deliverable.
- Назначай `area` обычно на этапе **Triage** (когда уже понятен тип работы). Если контекст ещё сырой — можно отложить.
- Создавай/используй `Area`, если есть повторяемые правила проверки/выполнения (например договоры, юридические проверки, финансы, QA).
- Пример: новые договоры по разным проектам, но одна методика проверки (риски исполнителя, реквизиты, условия) → задачи привязывай к `WA-contracts-and-legal`.
- Допустимо, что `area` у задачи отличается от `area` проекта, если задача относится к другому процессу (например проект дизайн, а задача — договорная проверка).
- При новых вводных агент сначала ищет подходящую существующую `Area`; новую `Area` создаёт только если подходящей нет и ожидается повторяемый процесс (не разовый кейс).
- Для заметок поле `area` опционально, но по умолчанию ставится, если заметка — рабочий референс/бриф/гайд в повторяемом контуре; для временного capture `area` можно отложить.
- Карточка `Area` (`*/40_Areas/*.md`) должна быть **индексным файлом**: назначение области, краткие правила/риски, краткие описания и ссылки на связанные заметки/референсы; длинный контент хранится в `*/50_Notes/`.

## Именование файлов (канонично)
- Личные: `P*`, рабочие: `W*` (чтобы `[[...]]` были уникальными).
- Задачи: `PT-<slug>-YYYY-MM-DD.md`, `WT-<slug>-YYYY-MM-DD.md`.
- Проекты: `PP-<slug>.md`, `WP-<slug>.md`.
- Области: `PA-<slug>.md`, `WA-<slug>.md`.
- Цели: `PG-<slug>.md`, `WG-<slug>.md`.
- Еженедельные обзоры: `weekly/YYYY-Www.md` внутри `70_Reviews/`.

## Шаблоны
Создавай новые сущности через `scripts/new-entity.sh` (предпочтительно) или через шаблоны из `00_System/Templates/`; заполняй frontmatter (минимум: `type`, `status`, `created`, `updated`).

## Правила заполнения Project и Task (по умолчанию)
- `Project` (`*/30_Projects/*.md`) — индексный файл результата: outcome, краткий контекст/риски, ссылки на ключевые заметки, автоподборки задач. Длинные брифы/сметы/конвертации храни в `*/50_Notes/`.
- `Task` (`*/20_Tasks/*.md`) — одна исполнимая единица (1 deliverable = 1 файл): человеческий заголовок, `Next step`, одна основная строка чекбокса, checklist, context/links.
- Не оставляй пустые поля `area/project/goal` в frontmatter; добавляй их, когда связь подтверждена.
- Длинные URL и подробные пояснения держи в `Context / Links`, а не в строке чекбокса (иначе перегружается Taskboard/Tasks-представления).

## Входящие → заметки → исполнение (правило по умолчанию)
- Для входящих материалов (DOCX/PPTX/аудио/PDF) сначала создай все необходимые заметки в `*/50_Notes/` и добавь `## Summary`.
- После этого задай уточняющие вопросы и дождись подтверждения пользователя.
- После подтверждения по умолчанию формируй **проект (`*/30_Projects/`) и минимум одну задачу (`*/20_Tasks/`)**, если входящее описывает новую работу/результат (а не чисто справочную заметку).

## Промпты ролей (если агент их поддерживает)
Ролевые промпты для “ассистента/коуча/мейнтейнера” лежат в `00_System/Prompts/`:
- `00_System/Prompts/Bootstrapper.md` (первый запуск)
- `00_System/Prompts/Assistant.md`
- `00_System/Prompts/Coach.md`
- `00_System/Prompts/Repo-Maintainer.md`
- `00_System/Prompts/Router.md` (выбор роли и декомпозиция)

## Совместимость с Tasks/Dataview
- В каждом файле задачи должен быть **хотя бы один** чекбокс `- [ ] ...` (Tasks‑плагин).
- Для планирования используй свойства Tasks (например, дедлайн/расписание) прямо в строке задачи.
- Чтобы запросы на страницах проекта/цели работали автоматически, **добавляй ссылку на проект/цель** в строку задачи (например: `... [[PP-...]] [[PG-...]]`).
- Для сводок по проектам/целям используй frontmatter + Dataview‑запросы.

## Обновления статусов
- Выполнено: отметь чекбокс как `- [x]` и (если есть) `status: done`.
- Заблокировано: добавь секцию `## Blockers` и выставь `status: waiting` (или `paused`).
- Завершённые/неактуальные: перенеси в соответствующий `99_Archive/` домена, если файл больше не нужен в активной навигации, и добавь `#no-graph`.
- По умолчанию архивирование делай на weekly review (пятница) или после closeout, а не сразу в момент первого `done` (если нет явного запроса пользователя).

## Коучинг (минимальный цикл)
- Цели держи в `60_Goals/` с метрикой/критериями успеха.
- Раз в неделю создай обзор в `70_Reviews/weekly/` (формат файла: `YYYY-Www.md`) и:
  - подведи итоги по задачам/проектам;
  - обнови план на следующую неделю;
  - скорректируй цели/приоритеты.
