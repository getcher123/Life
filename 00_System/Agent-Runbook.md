# Runbook для AI‑агента

Цель: поддерживать систему задач/заметок/проектов/целей так, чтобы она оставалась читаемой в Obsidian и аккуратной в Git.

## Первый запуск (bootstrap)
Если это первое подключение агента к репозиторию, выполни bootstrap по документу:
- `00_System/Bootstrap.md`
- промпт: `00_System/Prompts/Bootstrapper.md`

## Команды агента
Список команд и “ролей” для запросов: `00_System/Agent-Commands.md`.

## Единый форматный протокол
- Базовое правило: в vault каноника хранится в `.md`, бинарные форматы — входящие/артефакты.
- Для всех входящих кастомных форматов сначала делай `incoming -> note (50_Notes) -> Summary`.
- Только после подтверждения пользователя переходи к созданию `project/task` (если это рабочее входящее).
- Экспорт наружу делай из `.md` (например `md2docx`), чтобы не терялась история изменений.
- Полный стандарт форматов: `00_System/Conventions.md` (раздел `Форматы`).

## Старт агентской сессии (краткий протокол)
Перед выполнением запроса агенту рекомендуется:
1. Определить роль:
   - по явной команде пользователя (`@Assistant`, `@Router`, `@Repo-Maintainer`, и т.д.);
   - если роли нет — предложить релевантную роль и дождаться подтверждения.
2. Уточнить домен (`Work` / `Personal`), если это влияет на размещение файлов.
3. Уточнить ограничения:
   - что можно делать (`только заметка` / `создать задачи` / `без коммита`);
   - срок (`сегодня`, `до пятницы`, конкретная дата).
4. Если запрос про сводку/сроки:
   - уточнить текущую дату внутренними средствами агента;
   - помнить, что концом рабочей недели считается пятница.
5. После изменений задач/проектов:
   - синхронизировать `*/20_Tasks/00_Taskboard.md`, `*/30_Projects/00_Projects.md`, `*/20_Tasks/00_Kanban.md`.

Готовые команды запуска с примерами: `00_System/Agent-Commands.md`.

## Работа с DOCX (через скрипты)
Агент не “открывает” DOCX как UI-документ — вместо этого конвертирует DOCX ↔ Markdown через `pandoc`, чтобы читать/редактировать содержимое в vault.

### Общее правило (для всех входящих DOCX)
Если в запросе/входящих появляется документ в формате `.docx`, агент обязан:
1) создать в vault Markdown‑копию в `Work/50_Notes/` или `Personal/50_Notes/` и добавить `## Summary`;
2) задать 1–3 уточняющих вопроса пользователю;
3) только после подтверждения пользователя — создавать задачи/проекты и связывать их с заметкой;
4) по умолчанию (если это рабочее входящее на выполнение, а не чисто справочная заметка) — сформировать **и проект, и минимум одну задачу**.

Рекомендуемый способ (только заметка): `scripts/docx2note.sh "/path/to/input.docx"`
После подтверждения (заметка + задача): `scripts/docx2note.sh "/path/to/input.docx" --create-task --project [[WP-...]]`

1. Проверка `pandoc`:
   - если `pandoc` есть в `PATH`, можно сразу конвертировать;
   - иначе поставить локально в `.pandoc_local/`:
     - `scripts/setup-pandoc.sh 3.1.11`
     - (опционально) `source scripts/env-docx-tools.sh`
2. DOCX → Markdown (для чтения/анализа):
   - `scripts/docx2md.sh "/path/to/input.docx" "Work/50_Notes/<name>.md"`
2.1. DOCX → заметка (по умолчанию):
   - `scripts/docx2note.sh "/path/to/input.docx"`
2.2. DOCX → заметка + задача (только после подтверждения пользователя):
   - `scripts/docx2note.sh "/path/to/input.docx" --create-task --project [[WP-...]]`
3. Markdown → DOCX (если нужно вернуть документ):
   - `scripts/md2docx.sh "Work/50_Notes/<name>.md" "/path/to/output.docx"`
4. Правила:
   - не коммить `.pandoc_local/` (оно в `.gitignore`);
   - исходный DOCX не изменять; в vault хранить Markdown‑конвертацию как рабочую копию/выжимку.

## Работа с аудио (voice → text)
Если во входящих есть голосовое сообщение/аудиофайл (`.ogg/.mp3/.m4a/.wav`), агент:
1) расшифровывает в заметку `*/50_Notes/`;
2) добавляет в заметку секцию `## Summary` (1–5 буллетов: суть, сроки, варианты, следующий шаг);
3) задаёт 1–3 уточняющих вопроса пользователю;
4) только после подтверждения пользователя — создаёт задачи/проекты и связывает их с заметкой;
5) по умолчанию (если это входящее про новую работу/результат) — формирует **проект + задачу**.

Команда:
- `scripts/audio2note.sh "/path/to/audio.ogg"`
После подтверждения (заметка + задача): `scripts/audio2note.sh "/path/to/audio.ogg" --create-task --project [[WP-...]]`

## Работа с Google Sheets (service account)
Если в запросе есть ссылка на Google‑таблицу/нужно “свериться с таблицей заказов”, агент:
1) **не вставляет** JSON сервисного аккаунта в заметки/репозиторий;
2) читает таблицу через `scripts/gsheets_fetch.py` (см. `00_System/Google-Sheets-Tools.md`);
3) по умолчанию делает только чтение/проверку/сводку; создание задач/проектов — после подтверждения пользователя.

## Режимы работы агента (выбери один на запрос)
Если агент поддерживает “роли”, используй промпты из `00_System/Prompts/` (Router/Assistant/Coach/Repo‑Maintainer).

### A) Capture (быстрый захват)
1. Создай файл в `Work/00_Inbox/` или `Personal/00_Inbox/` с кратким заголовком.
2. Не оптимизируй структуру — только зафиксируй входящее.
3. (Опционально) Добавь 1–3 уточняющих вопроса в конец файла.

### B) Triage (разбор Inbox)
1. Просмотри новые файлы в `*/00_Inbox/`.
2. Для каждого элемента выбери исход:
   - задача → `*/20_Tasks/` (1 файл = 1 задача);
   - проект → `*/30_Projects/`;
   - заметка → `*/50_Notes/`;
   - цель → `*/60_Goals/`;
   - мусор/неактуально → удалить только по явному запросу, иначе архивировать в `*/99_Archive/`.
3. Определи `area` (если уже понятен тип работы):
   - `area` ставь для повторяемых процессов/зон ответственности (гайдлайны, чеклисты, стандарты), а не для разового outcome;
   - если контекст сырой — отложи назначение `area` до уточнений;
   - допустимо, что `area` у задачи отличается от `area` проекта (например задача — юридическая проверка по проекту дизайна).
   - сначала проверь существующие `*/40_Areas/` и подбери подходящую `Area`;
   - если совпадение неочевидно (несколько вариантов) — предложи пользователю выбор;
   - создавай новую `Area` только если подходящей нет и виден повторяемый процесс/гайдлайн (не разовый кейс).
   - для заметок (`*/50_Notes/`) ставь `area`, если заметка будет переиспользуемым референсом/брифом/гайдом в этом контуре; если это временный capture и `Area` неясна — оставь без `area`.
4. В исходном inbox‑файле оставь ссылку `[[...]]` на созданный “канонический” файл (или перенеси содержимое и оставь stub).

### C) Plan (планирование)
1. Уточни контекст: домен (work/personal), горизонты (сегодня/неделя/квартал), ограничения.
2. Создай/обнови:
   - цели (если это про направление) в `*/60_Goals/`;
   - проекты (если это про результат) в `*/30_Projects/`;
   - задачи (если это про действия) в `*/20_Tasks/`.
   - проект веди как индекс результата (outcome + ссылки на заметки + автоподборки задач), длинные материалы выноси в `*/50_Notes/`;
   - задачу веди как одну исполнимую единицу (1 deliverable) с `Next step` + основной чекбокс + checklist.
3. Для задач проекта/цели: добавь в строку задачи ссылки `[[WP-...]]` / `[[WG-...]]` (или personal‑аналог).
4. Для задач с датой: используй свойства Tasks в строке (`📅`, `⏳`, `🛫`).
5. Если задачи создаются в повторяемом контуре (договоры, финансы, QA и т.п.) — назначай `area` по процессу/гайдлайну, а не только по теме проекта.
6. Если при планировании появляется новый повторяемый контур, которого нет в `*/40_Areas/`, предложи создать новую `Area` (с кратким описанием правил/чеклиста).

### D) Coach (коучинг по целям)
1. Открой файл цели `*/60_Goals/...` и проверь:
   - “Почему важно”, “метрика”, “горизонт”.
2. Уточни 1 следующий шаг: какой проект/задача двигает метрику сильнее всего.
3. Обнови план (добавь/переприоритизируй задачи) и предложи ритм ревью (weekly).

## Качество данных (обязательно)
- Не делай массовых переименований/переформатирования.
- Не удаляй информацию без запроса: используй `status` и/или `99_Archive/`.
- При архивировании проекта архивируй связанные заметки по полю `project:` (если заметка ссылается на архивируемый проект).
- При архивировании проекта проверяй связанные цели (`goal:` в проекте): если цель больше не связана с активными проектами (и не используется как активная рабочая цель), архивируй и её.
- Для новых сущностей соблюдай язык тех-ID/slug из `00_System/Vault-Config.env` (`VAULT_TECH_ID_LANGUAGE`); текущие файлы не переименовывай без отдельного запроса.
- Для ручного создания новых сущностей используй `scripts/new-entity.sh` (он читает `VAULT_TECH_ID_LANGUAGE` и снижает риск англоязычных slug при режиме `русский`).
- Карточку `Area` веди как **индексный файл**:
  - назначение области + краткие правила/чеклист/риски;
  - краткие summary и ссылки на связанные заметки/референсы;
  - не складируй в `Area` длинные брифы/сметы/расшифровки (они должны жить в `*/50_Notes/`).
- Карточку `Project` веди как **индекс результата**, а не как длинную рабочую заметку:
  - outcome / краткие ограничения / ссылки на ключевые notes / автоподборки задач;
  - длинные брифы, сметы, конвертации и исследования — в `*/50_Notes/`.
- Карточку `Task` веди как **исполнимую единицу**:
  - одна основная строка чекбокса (`deliverable`) + `Next step` + checklist;
  - длинные ссылки/тексты — в `Context / Links`, не в строке задачи.
- В каждой задаче должен быть хотя бы один чекбокс `- [ ]`.
- Обновляй `updated: YYYY-MM-DD` при существенных изменениях.

## Триггеры архивирования (по умолчанию)
Архивирование — это отдельный этап **после выполнения/closeout**, а не автоматическое действие в момент первого `done`.

### Этапы workflow (коротко)
1) `Incoming -> Notes` (входящие в заметки)  
2) `Notes -> Project + Task` (после подтверждения)  
3) `Execution` (работа по задачам)  
4) `Closeout` (deliverable сделан/отправлен/принят, закрытие хвостов)  
5) `Archive` (перенос из активной навигации в `99_Archive/`)

### Когда архивировать задачу (default trigger)
Архивируй задачу, если одновременно:
- основной deliverable-чекбокс выполнен (`- [x]`);
- нет незакрытых пунктов в `Checklist` (или они явно перенесены/отменены);
- нет активных блокеров/ожиданий (`status: waiting`, незавершённый `Blockers`);
- задача больше не нужна в ближайшей активной навигации/контроле;
- по умолчанию прошёл минимум **1 weekly review** после завершения (буфер от поздних правок).

Исключения (не архивировать сразу):
- ожидаются правки от заказчика;
- не закрыты финансовые/закрывающие хвосты, если они относятся к этой же задаче;
- это контрольная/мониторинговая задача (follow-up);
- пользователь явно просит оставить задачу в активных.

### Когда архивировать проект (default trigger)
Архивируй проект, если одновременно:
- `Outcome` достигнут **или** проект отменён (`status: done/canceled`);
- нет активных задач проекта (или они перенесены/перепривязаны);
- ключевые заметки/артефакты сохранены и связаны (`brief`, смета, ТЗ, источники);
- нет подтверждённого ближайшего продолжения в том же scope (иначе лучше `paused`).

Рекомендуемый момент:
- основной триггер — **weekly review (пятница)** или отдельный closeout review по проекту.

### Что архивировать вместе с проектом (цепочка)
При архивировании проекта:
1. Перенести сам проект в `*/99_Archive/30_Projects/`.
2. Перенести связанные завершённые задачи (если относятся только к этому проекту и не нужны в активной навигации).
3. Перенести связанные заметки по полю `project:`.
4. Проверить связанную цель (`goal:`):
   - архивировать цель только если у неё не осталось активных проектов и она не используется как общая активная цель.

## Авто‑разбор входящих материалов (drop folder)
Для входящих файлов (DOCX/аудио) используем папку‑буфер:
- `Work/00_Inbox/Incoming/` (файлы не коммитятся в Git)

Цикл:
1) пользователь кладёт файлы в `Work/00_Inbox/Incoming/`;
2) агент запускает `scripts/process-incoming.sh --domain work`:
   - создаются заметки в `Work/50_Notes/` (с `## Summary`) из DOCX/PPTX/аудио;
   - исходники перемещаются в `Work/99_Archive/Incoming/` (вместо удаления);
3) агент задаёт 1–3 уточняющих вопроса по каждой заметке;
4) только после подтверждения пользователя создаёт задачи/проекты и связывает их с заметкой;
5) по умолчанию для рабочих входящих — создаёт **проект и минимум одну задачу** (если пользователь не попросил режим `только заметка`).

## Мини‑протокол Git (если агент коммитит)
1. `git status` → проверь, что меняется только ожидаемое.
2. `git add -A`
3. `git commit -m "<коротко: что сделано>"`
