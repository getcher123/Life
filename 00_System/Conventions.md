# Конвенции

## Общие принципы
- Markdown‑first: `.md` + вложения в `90_Assets/`.
- Стабильные пути: избегай массовых переименований/перемещений (важна Git‑история).
- Ссылки: используй `[[...]]` между сущностями.

## Форматы (каноничное правило)
- Канонический рабочий формат vault: **Markdown (`.md`)**.
- Сущности системы (`Tasks/Projects/Areas/Goals/Reviews/Notes`) храним как `.md`.
- Не‑md форматы (DOCX/PPTX/PDF/аудио/XLSX/изображения) считаются **входящими или артефактами**, а не канонической формой.

### Входящие кастомные форматы (ingest)
- Временная точка входа: `*/00_Inbox/Incoming/`.
- Приватные/чувствительные входящие: `*/00_Inbox/Private/`.
- После разбора:
  1) создать заметку в `*/50_Notes/` с `## Summary`;
  2) связать источник/артефакт ссылками;
  3) исходник переместить в `*/99_Archive/Incoming/` (или в `*/00_Inbox/Private/`, если нужен приватный доступ).

### Правило конвертации по типам
- DOCX -> MD: `scripts/docx2note.sh` / `scripts/docx2md.sh`
- PPTX -> MD: `scripts/pptx2note.sh`
- Аудио -> MD: `scripts/audio2note.sh` (или skill `openai-audio-to-markdown`)
- PDF -> заметка + артефакты (см. `00_System/pdf-tool.md`)
- XLSX/CSV/JSON -> заметка-выжимка + ссылка на источник/снапшот
- Изображения/медиа -> хранить в `*/90_Assets/` и ссылаться из `.md`

### Экспорт форматов (outbound)
- Для передачи наружу экспортируем **из канонического `.md`**:
  - MD -> DOCX: `scripts/md2docx.sh`
  - при необходимости MD -> PDF через pandoc/toolchain проекта
- Экспортные файлы по умолчанию складываем в:
  - `*/50_Notes/Exports/` (если это документы/выгрузки заметок),
  - либо рядом с целевым проектным артефактом в `*/90_Assets/`.

### Анти-дрейф форматов
- Не редактировать “боевые” документы только в бинарном формате: изменения сначала фиксировать в `.md`.
- В `.md` держать ссылку на исходник и экспорт.
- Новые задачи/проекты создавать после цикла: `incoming -> notes -> confirmation -> task/project`.

## Язык технических ID / slug (имена новых файлов)
- Глобальная настройка vault: `00_System/Vault-Config.env`
- Переменная: `VAULT_TECH_ID_LANGUAGE`
- Допустимые значения:
  - `русский` — новые имена файлов могут содержать русскоязычные слова (кириллица в slug)
  - `english` — ASCII/латиница (транслитерация)
- Для этого хранилища установлено: `VAULT_TECH_ID_LANGUAGE=русский`
- Важно: текущие сущности **не переименовываются автоматически**; настройка влияет только на новые сущности/файлы.

## Frontmatter (минимальный контракт)
Рекомендуемые поля (YAML):
- `type`: `task | project | area | goal | review | note`
- `status`: `todo | doing | waiting | active | paused | done | canceled`
- `created`, `updated`: `YYYY-MM-DD`
- `area`, `project`, `goal`: `[[...]]` (при наличии)
- `tags`: список
- Не держи пустые поля (`area:`, `goal:`, `project:`) без значения — лучше добавить их позже, когда связь подтверждена.

## Семантика `area` (важно для triage)
- `area` — это не “что нужно завершить”, а “в каком постоянном контуре ответственности/стандартов выполняется работа”.
- Ставь `area`, когда понятен повторяемый процесс/гайдлайн (юридическая проверка, финансы, операционка, QA и т.п.).
- Если контекст ещё неясен, `area` можно не ставить сразу и заполнить после уточнений.
- У задачи `area` может отличаться от `area` проекта, если задача относится к отдельному процессу.
- Для заметок `area` — **опционально**, но по умолчанию ставится, если заметка является рабочим референсом/брифом/гайдом/источником в повторяемом контуре.
- Для проектных заметок (brief, оценка, ТЗ, референсы по проекту) обычно полезно ставить и `project:`, и `area:` одновременно.
- Если заметка разовая/временная и `Area` неочевидна — оставь без `area`, заполни после triage.
- Пример: договоры разных проектов проверяются по одной методике → задачи проверки договора относятся к `WA-contracts-and-legal`.
- Policy для агента при новых вводных:
  - сначала подобрать существующую `Area`;
  - если есть 2–3 близкие — предложить выбор пользователю;
  - новую `Area` создавать только если подходящей нет и это повторяемый процесс/контур, а не разовая задача.

## Карточка `Area` = индексный файл (правило заполнения)
- Файл `*/40_Areas/PA-...` / `WA-...` — это **индекс/карта области**, а не место хранения длинных рабочих материалов.
- В `Area`-файле держим:
  - `Definition` (назначение области);
  - краткие правила/чеклисты/риски (если есть);
  - **краткие описания + ссылки** на связанные заметки/референсы;
  - автоматические списки проектов (Dataview) и при необходимости связанные сущности.
- Подробный контент (брифы, сметы, конвертации DOCX/PPTX, аналитика) хранится в `*/50_Notes/`, а в `Area` остаются ссылки на него.
- При обновлении `Area` избегай копирования длинных текстов из заметок; добавляй 1–2 строки summary и ссылку.

## Карточка `Project` = индекс результата (правило заполнения)
- Файл `*/30_Projects/PP-...` / `WP-...` — это **карточка результата (outcome)** и индекс проекта, а не место для длинных брифов/расшифровок.
- В `Project`-файле по умолчанию держим:
  - `Outcome` (что считается “готово”);
  - краткий контекст/ограничения/риски;
  - краткие ссылки на ключевые заметки (`brief`, `estimate`, `source docs`, `handoff`);
  - автоподборки задач (open/done) через Tasks/Dataview.
- Подробный контент проекта (DOCX-конвертации, длинные брифы, сметы, исследования) хранится в `*/50_Notes/`.
- В `Project` добавляй `area:` и `goal:`, когда они понятны и реально помогают навигации; не оставляй пустыми.
- Если проект затрагивает несколько контуров, у проекта держи одну **primary area**, а дополнительные отражай через `task.area` и/или секцию `Related areas`.

## Карточка `Task` = исполнимая единица (правило заполнения)
- Файл `*/20_Tasks/PT-...` / `WT-...` — это **одна исполнимая задача / deliverable**.
- По умолчанию: `1 файл задачи = 1 deliverable = 1 основная строка чекбокса`.
- Если внутри много подшагов, используй `## Checklist`, а не создавай пачку мелких файлов задач.
- Создавай отдельные task-файлы только если отличаются:
  - дедлайн;
  - ответственный;
  - статус/поток работы;
  - независимое выполнение.
- В задаче должны быть:
  - человеческий заголовок (`# ...`);
  - `## Next step` (один ближайший шаг);
  - одна основная строка `- [ ] ...` (deliverable);
  - ссылки на проект/цель в строке задачи (если релевантно) для автоподборок;
  - `## Context / Links` (источники, brief, note) — длинные URL/описания держать здесь, а не в строке чекбокса.
- `Task`-файл — не место для длинного ТЗ: длинные материалы держи в `*/50_Notes/`, а в задаче оставляй ссылку и короткий контекст.

## Tasks plugin: свойства в строке задачи
Используй свойства Tasks прямо в чекбоксе (примерная схема):
- Дедлайн: `📅 YYYY-MM-DD`
- Запланировано: `⏳ YYYY-MM-DD`
- Старт: `🛫 YYYY-MM-DD`
- Завершено: `✅ YYYY-MM-DD`

### Привязка задач к проектам/целям
- Для задач проекта/цели добавляй в строку задачи ссылку на соответствующий файл: `[[PP-...]]`, `[[WP-...]]`, `[[PG-...]]`, `[[WG-...]]`.
- В шаблонах проектов/целей используются плейсхолдеры Tasks `{{query.file.filenameWithoutExtension}}`, поэтому не нужно вручную подставлять имя проекта/цели в запросах.
- Архивирование связанных сущностей:
  - при архивировании проекта переноси связанные заметки по полю `project:`;
  - проверяй связанную цель по полю `goal:` и архивируй её только если у неё не остаётся активных проектов.
- Триггеры архивирования (default):
  - задачу архивировать не сразу после `done`, а обычно после `weekly review` (когда нет хвостов/ожиданий);
  - проект архивировать после closeout: outcome достигнут (или canceled), активных задач не осталось, ключевые notes связаны.

Храни каноническую задачу в файле из `20_Tasks/`, а ссылки на неё — в проектах/обзорах/дневниках.
