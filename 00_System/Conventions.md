# Конвенции

## Общие принципы
- Markdown‑first: `.md` + вложения в `90_Assets/`.
- Стабильные пути: избегай массовых переименований/перемещений (важна Git‑история).
- Ссылки: используй `[[...]]` между сущностями.

## Язык технических ID / slug (имена новых файлов)
- Глобальная настройка vault: `00_System/Vault-Config.env`
- Переменная: `VAULT_TECH_ID_LANGUAGE`
- Допустимые значения:
  - `русский` — новые имена файлов могут содержать русскоязычные слова (кириллица в slug)
  - `english` — ASCII/латиница (транслитерация)
- Для этого хранилища установлено: `VAULT_TECH_ID_LANGUAGE=русский`
- Важно: текущие сущности **не переименовываются автоматически**; настройка влияет только на новые сущности/файлы.

## Frontmatter (минимальный контракт)
Рекомендуемые поля (YAML):
- `type`: `task | project | area | goal | review | note`
- `status`: `todo | doing | waiting | active | paused | done | canceled`
- `created`, `updated`: `YYYY-MM-DD`
- `area`, `project`, `goal`: `[[...]]` (при наличии)
- `tags`: список

## Tasks plugin: свойства в строке задачи
Используй свойства Tasks прямо в чекбоксе (примерная схема):
- Дедлайн: `📅 YYYY-MM-DD`
- Запланировано: `⏳ YYYY-MM-DD`
- Старт: `🛫 YYYY-MM-DD`
- Завершено: `✅ YYYY-MM-DD`

### Привязка задач к проектам/целям
- Для задач проекта/цели добавляй в строку задачи ссылку на соответствующий файл: `[[PP-...]]`, `[[WP-...]]`, `[[PG-...]]`, `[[WG-...]]`.
- В шаблонах проектов/целей используются плейсхолдеры Tasks `{{query.file.filenameWithoutExtension}}`, поэтому не нужно вручную подставлять имя проекта/цели в запросах.

Храни каноническую задачу в файле из `20_Tasks/`, а ссылки на неё — в проектах/обзорах/дневниках.
